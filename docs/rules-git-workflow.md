# Git / 브랜치 / 커밋 / PR 규칙

> 이 문서는 실제 커밋/브랜치/PR 작업을 할 때 참조한다.

## 브랜치 전략

- `main` — 운영 배포 기준 브랜치
- `develop` — 통합 브랜치
- `feature/이슈번호-설명` — 기능 개발
- `hotfix/설명` — 긴급 수정

`main` 브랜치 직접 커밋은 금지한다. 모든 변경은 PR을 통해서만 반영한다.

## 커밋 메시지

- 이슈 번호를 포함한다. 예: `[#12] 장바구니 수량 변경 API 추가`
- 하나의 커밋은 하나의 논리적 변경 단위로 유지한다.

## PR 규칙

- 이슈 1개당 PR 1개
- PR 설명에는 최소한 다음을 포함한다.
  - 관련 이슈 번호
  - 변경 파일 목록 및 변경 이유
  - 테스트 방법 및 결과
  - 스크린샷/재현 절차 (UI 변경 시)
  - 롤백 방법
- 변경 파일 10개 이내 / diff 400줄 이내를 넘으면 분할을 우선 검토한다.
