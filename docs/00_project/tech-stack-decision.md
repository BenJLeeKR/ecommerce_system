# 기술 스택 결정 문서 (Tech Stack Decision)

> 상태: 초안 (Draft) — 최종 확정 전 설계 승인자(기획자) 검토 필요
> 관련 문서: `docs/00_project/project-scope.md` (MVP 우선, 단계적 확장 원칙)

---

## 1. 배경 및 목적

본 프로젝트는 다음 조건 하에서 기술 스택을 선택한다.

- 개발은 Claude Code(AI 에이전트)가 수행하고, 기획자는 계획/설계를 승인하고 결과물을 검수한다.
- 기획자의 **코드 리뷰 능력이 높지 않으므로**, 리뷰 부담이 적은 스택을 우선한다.
- `project-scope.md`에서 정한 대로 **MVP부터 단계적으로 확장**하는 방식을 따른다. 즉, 오늘의 선택 기준은 "이론상 최고 확장성"이 아니라 "MVP를 안전하고 빠르게 만들고, 이후 단계적으로 키울 수 있는가"이다.

비교 대상은 다음 두 안이다.

| | 안 A | 안 B |
|---|---|---|
| Frontend | Next.js + TypeScript | Next.js + TypeScript |
| Backend | NestJS + TypeScript | Spring Boot (Java) |
| DB | PostgreSQL | PostgreSQL |
| ORM | Prisma | JPA (Hibernate) |
| UI | Tailwind CSS + shadcn/ui | Tailwind CSS + shadcn/ui |
| Test | Jest + Playwright | JUnit + Playwright |
| Deploy | Docker + GitHub Actions | Docker + GitHub Actions |

두 안의 핵심 차이는 **백엔드 언어(TypeScript vs Java)와 그에 따른 생태계**이며, 프론트엔드/DB/UI/배포는 동일하다. 따라서 아래 비교는 사실상 "백엔드를 어떤 언어·프레임워크로 할 것인가"의 비교다.

## 2. 비교 기준별 분석

### 2-1. AI 에이전트(Claude Code) 개발 적합성

| | 안 A | 안 B |
|---|---|---|
| 평가 | 프론트-백엔드가 동일 언어(TypeScript)라 Claude Code가 하나의 문맥(타입, 문법)으로 작업 가능. Prisma의 `schema.prisma`는 선언적이고 짧아서 AI가 스키마 변경 계획을 세우기 쉬움. NestJS는 모듈/데코레이터 구조가 강제되어 있어 AI가 정해진 패턴을 벗어나기 어려움(=일관성 확보에 유리) | Java/Spring Boot도 학습 데이터가 매우 방대하고 관례(convention)가 강해 AI가 패턴을 따르기 쉬움. 다만 Controller/Service/Repository/Entity/DTO/Mapper 등 계층이 많아 기능 하나당 만져야 할 파일 수가 A보다 많고, 프론트(TS)와 백엔드(Java) 두 언어를 오가야 함 |
| 결론 | **A 우세**. 언어 통일 + 파일 수가 적어 AI가 "요청 범위 밖 파일을 건드릴 위험"도 상대적으로 낮음 (절대 규칙 3과 결이 맞음) | |

### 2-2. 대규모 이커머스 확장성

| | 안 A | 안 B |
|---|---|---|
| 평가 | Node.js는 I/O 위주 작업(DB 조회, 외부 API 호출 등 이커머스의 대부분 트래픽 패턴)에 강하고, 수평 확장(서버를 여러 대로 늘리는 방식)으로 충분히 대규모 트래픽을 감당한 사례가 많음(Netflix, PayPal 등) | JVM/Spring은 복잡한 트랜잭션 처리, 대규모 배치 작업, 엔터프라이즈급 통합(메시지 큐, 대량 데이터 처리 등)에 검증된 사례가 압도적으로 많음. 국내 대형 이커머스/금융권 상당수가 Java/Spring 기반 |
| 결론 | 초기~중형 규모 트래픽에는 A로 충분. 다만 매우 복잡한 트랜잭션/배치/대규모 조직 협업이 필요한 수준까지 가면 **B가 이론적으로 더 검증된 선택지**. 단, 이는 "지금 당장"이 아니라 "수년 뒤 초대형 규모가 됐을 때"의 이야기 |

### 2-3. 기획자가 검수하기 쉬운 정도

| | 안 A | 안 B |
|---|---|---|
| 평가 | 언어가 하나(TypeScript)라 앞뒤 코드를 오갈 때 문맥 전환 부담이 적음. `schema.prisma`는 애노테이션이 적고 "테이블에 어떤 컬럼이 있는지"가 한눈에 보이는 짧은 선언 파일이라 비개발자도 어느 정도 읽을 수 있음 | JPA 엔티티는 애노테이션(`@Entity`, `@Column`, `@OneToMany` 등)이 많아 비개발자가 훑어보기엔 상대적으로 어렵고, 프론트(TS)·백엔드(Java) 두 문법을 오가야 해서 검수 시 인지 부담이 커짐 |
| 결론 | **A 우세**. 사용자님이 명시한 "코드 리뷰 능력이 높지 않다"는 조건에서 가장 크게 갈리는 기준 |

### 2-4. 테스트 자동화 용이성

| | 안 A | 안 B |
|---|---|---|
| 평가 | Jest는 설정이 간단하고 NestJS 자체에 테스트 유틸리티(TestingModule)가 내장되어 있어 단위/통합 테스트 작성이 쉬움. E2E는 Playwright로 동일 | JUnit5 + Spring Boot Test(`@SpringBootTest`, `@DataJpaTest`, `@WebMvcTest` 등)는 "레이어별로 끊어서 테스트"하는 구조가 매우 성숙하고 표준화되어 있음. E2E는 Playwright로 동일(프론트 동일하므로) |
| 결론 | 두 안 모두 우수. 근소하게 **B의 백엔드 테스트 계층 구조가 더 정교**하지만, A도 실무적으로 충분히 검증된 수준 → **실질적으로 우열 없음(근소하게 B)** |

### 2-5. 운영 안정성

| | 안 A | 안 B |
|---|---|---|
| 평가 | Node.js도 프로덕션에서 널리 검증됨. 다만 헬스체크/메트릭 등 운영 도구는 커뮤니티 패키지(`@nestjs/terminus` 등)에 의존 | Spring Boot는 Actuator가 기본 내장되어 헬스체크·메트릭·모니터링 연동이 표준화되어 있고, JVM은 수십 년간 대규모 운영 노하우가 축적된 플랫폼 |
| 결론 | **B가 근소 우세**. 다만 A도 운영 불가능한 수준은 아니며, 커뮤니티 도구로 충분히 보완 가능 |

### 2-6. 개발 속도

| | 안 A | 안 B |
|---|---|---|
| 평가 | 언어 통일 + Prisma의 빠른 스키마-클라이언트 생성 + Node의 빠른 재시작(hot reload)으로 이커머스에 많은 CRUD성 기능을 빠르게 만들기 유리 | 계층(Controller/Service/Repository/Entity/DTO)이 많아 기능 하나 추가 시 건드릴 파일이 늘어나고, Java 컴파일 과정이 있어 반복 개발 주기가 A보다 다소 느림 |
| 결론 | **A 우세**. "작은 단위로 여러 번 반복"하는 이번 프로젝트의 작업 방식과도 잘 맞음 |

### 2-7. 장기 유지보수성

| | 안 A | 안 B |
|---|---|---|
| 평가 | NestJS는 Node 생태계 중에서는 구조가 잡혀있어 상대적으로 안정적이지만, JS/Node 생태계 전반은 패키지 변경/deprecated 주기가 Java 진영보다 빠른 편 | Java/Spring은 하위 호환성을 매우 중요하게 여기는 생태계라 수년~10년 단위로 유지보수할 때 변경 폭이 작은 편 |
| 결론 | **B가 근소 우세**. 다만 NestJS 자체의 구조화된 설계가 이 격차를 상당 부분 줄여줌 |

## 4. 최종 추천안: 안 A (Next.js + NestJS + TypeScript + Prisma)

### 추천 근거

1. 이번 프로젝트의 개발 주체가 **Claude Code(AI 에이전트)**이고, 검수 주체가 **코드 리뷰에 익숙하지 않은 기획자**라는 조건이 명시적으로 주어졌다. 이 두 조건에서 가장 크게 갈리는 기준(1. AI 적합성, 3. 검수 용이성)에서 안 A가 뚜렷하게 앞선다.
2. `project-scope.md`에서 정한 "MVP부터 단계적으로" 원칙과, 개발 속도(6번 기준)에서 안 A가 유리한 점이 맞아떨어진다.
3. 안 B가 우세한 항목(운영 안정성, 장기 유지보수성, 초대형 확장성)은 모두 "근소한 우세" 또는 "먼 미래 초대형 규모"를 전제로 한다. 지금 시점에 이 이유만으로 리뷰 부담과 개발 속도를 희생할 근거는 약하다.

### 최종 스택

- Frontend: Next.js + TypeScript
- Backend: NestJS + TypeScript
- DB: PostgreSQL
- ORM: Prisma
- UI: Tailwind CSS + shadcn/ui
- Test: Jest + Playwright
- Deploy: Docker + GitHub Actions

## 5. 선택하지 않은 안(B)의 단점 (명확히 표기)

안 B(Spring Boot)를 채택하지 않는 이유와 단점을 명확히 정리한다.

- **검수 부담이 크다**: JPA 엔티티의 애노테이션 기반 문법과 계층 구조는 코드 리뷰에 익숙하지 않은 기획자가 훑어보기 어렵다. 프론트(TS)·백엔드(Java) 두 언어를 오가며 검수해야 하는 부담도 추가된다.
- **AI 에이전트 작업 시 파일 수 증가**: 기능 하나를 추가할 때 Controller, Service, Repository, Entity, DTO, Mapper 등 여러 파일을 동시에 만져야 해서, "요청 범위 밖 파일을 건드리지 않는다"는 절대 규칙을 지키기가 상대적으로 더 까다롭다.
- **개발 속도가 상대적으로 느리다**: Java 컴파일 과정과 많은 보일러플레이트로 인해 작은 단위 반복 개발(이번 프로젝트의 핵심 작업 방식) 주기가 A보다 길어진다.
- **초기 학습/설정 난이도**: Spring Boot의 설정 방식(빈 등록, 설정 파일 구조 등)은 개념이 많아, 비개발자 기획자가 설계 문서를 이해하는 데 필요한 배경지식이 A보다 많다.
- **장점이 필요해지는 시점이 아직 아님**: B가 앞서는 운영 안정성·장기 유지보수성·초대형 확장성은 실제로 트래픽/조직 규모가 커졌을 때 체감되는 장점인데, 이번 프로젝트는 MVP부터 단계적으로 검증하며 키우는 방식이라 그 장점을 지금 당장 활용하기 어렵다.

## 6. 선택한 안(A)의 한계와 대응 방안

정직한 판단을 위해 안 A의 한계도 함께 기록한다.

| 한계 | 대응 방안 |
|---|---|
| Node.js는 CPU 연산이 무거운 작업(대규모 배치, 복잡한 계산)에는 상대적으로 불리 | 이커머스 MVP~중형 규모에서는 대부분 I/O 위주 작업이라 문제되지 않음. 향후 무거운 배치가 필요해지면 해당 기능만 별도 서비스로 분리하는 방안을 검토(전체 재작성 아님) |
| JS/Node 생태계의 패키지 변경 주기가 빠른 편 | NestJS처럼 구조화된 프레임워크를 사용해 변경 영향을 최소화하고, 의존성 업데이트를 정기적으로 작은 단위 PR로 관리 |
| Spring Boot 대비 운영 모니터링 도구가 기본 내장이 아님 | `@nestjs/terminus` 등 커뮤니티 표준 패키지로 헬스체크/메트릭을 보완 (운영 단계 이슈로 별도 관리) |
| 정말 초대형 규모(트래픽/조직)로 성장 시, 일부 재검토 필요할 수 있음 | 지금 결정은 "돌이킬 수 없는 결정"이 아니라 "MVP~중형 규모까지의 결정"으로 간주하고, 향후 필요 시 특정 도메인(예: 결제/정산)만 별도 서비스로 분리하는 확장 경로를 열어둔다 |
