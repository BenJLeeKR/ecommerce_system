# 프로젝트 범위 정의서 (Project Scope)

> 이 문서는 개발자가 아니어도 이해할 수 있도록 작성되었다. 아직 설계·구현 단계가 아니며, **"무엇을 만들고 무엇을 만들지 않을 것인지"를 합의하기 위한 문서**다.
> 문서 안의 `➕ 제안` 표시는 기획자가 원래 생각한 범위에 Claude가 검토 후 추가·조정을 제안한 부분이다. 최종 채택 여부는 기획자가 결정한다.

---

## 1. 프로젝트 목표

- 자사 단일 쇼핑몰(직영몰) 형태의 이커머스 서비스를 구축한다.
- 처음부터 모든 기능을 만들지 않고, **최소 기능(MVP)으로 먼저 서비스가 굴러가는 상태**를 만든 뒤, 실제 운영 데이터를 보며 단계적으로 기능을 확장한다.
- 개발은 Claude Code가 담당하고, 기획자는 요구사항 정의·설계 승인·검수·최종 의사결정을 담당한다. (별도 작업 원칙 문서 참고)
- 특히 **주문/결제/재고**는 실수 시 금전적 손실이나 신뢰 문제로 직결되므로, 속도보다 정확성과 검증을 우선한다.

---

## 2. 1차 MVP 범위

MVP는 "고객이 상품을 보고, 장바구니에 담고, 주문하고, 결제하고, 관리자가 그 주문을 처리할 수 있는" 최소한의 흐름이 끊김 없이 동작하는 것을 목표로 한다.

| 영역 | 항목 | 비고 |
|---|---|---|
| 회원 | 회원가입, 로그인/로그아웃 | 원안 유지 |
| 회원 | 비밀번호 재설정 | ➕ 제안 — 실서비스 운영 시 필수적인 최소 안전장치 |
| 관리자 계정 | 관리자 로그인 | ➕ 제안 — 고객 로그인과 분리된 별도 인증 |
| 상품 | 상품 목록, 상품 상세, 카테고리 | 원안 유지 |
| 상품 | 기본 키워드 검색 (상품명 검색) | ➕ 검토 필요 — 카테고리 수가 적으면 MVP에서 생략 가능. 상품 수가 많다면 포함 권장 |
| 장바구니 | 장바구니 담기/수정/삭제 | 원안 유지 |
| 주문 🔴 | 주문 생성, 주문 조회 | 위험 영역 (4장 참고) |
| 주문 🔴 | 주문 취소 (배송 시작 전 단순 취소만) | ➕ 검토 필요 — 실서비스 최소 요건이지만 결제 취소 로직과 맞물려 복잡도 상승 |
| 재고 🔴 | 재고 차감 (주문 시 자동 차감) | 위험 영역 (4장 참고) |
| 결제 🔴 | 결제 요청/승인/실패 처리 | 위험 영역 (4장 참고), 단일 PG사만 우선 연동 |
| 배송 | 주문 시 배송지 입력 | ➕ 제안 — "배송" 흐름이 핵심 업무에 포함되어 있으므로 최소 기능 필요 |
| 배송 | 관리자의 배송상태 수동 변경 (준비중→배송중→배송완료) | ➕ 제안 — 실제 택배사 API 연동 없이 수동 관리로 시작 |
| 관리자 | 상품 등록/수정/삭제, 재고 수량 관리 | 원안 유지 |
| 관리자 | 주문 목록 확인, 주문 상태 변경 | 원안 유지 |
| 정책(비기능) | 이용약관/개인정보처리방침 동의 화면 | ➕ 제안 — 기능은 아니지만 서비스 오픈 전 법적으로 필요 |

> `➕ 제안` 항목은 개발 범위를 키우는 게 아니라, "실제로 서비스를 열 수 있는 상태"를 만들기 위한 최소 보완이다. 기획자가 판단해 제외해도 무방한 항목은 "검토 필요"로 표시했다.

## 3. 2차 고도화 범위

MVP로 서비스를 운영해본 뒤, 데이터와 사용자 반응을 보고 순서를 정해 추가한다. 아래는 우선순위가 정해진 목록이 아니라 후보 목록이다.

- 쿠폰
- 포인트
- 회원 등급제
- 상품 리뷰
- 추천 상품
- 검색 고도화 (오타 허용, 인기순/연관도 정렬 등)
- 주문 부분취소/부분환불, 교환
- 관리자 권한 세분화 (관리자 / 운영자 분리, 5장 참고)
- 다양한 결제수단 추가 (간편결제 등)
- 주문/배송 알림 (이메일, 알림톡 등)

---

## 4. 이번 프로젝트에서 제외할 범위

아래 항목은 "나중에 할 기능"이 아니라, **이번 프로젝트의 사업 모델(단일 직영몰) 범위를 벗어나는 항목**으로 판단하여 제외를 제안한다. 필요성이 생기면 이번 프로젝트의 연장이 아니라 **별도 프로젝트**로 다루는 것을 권장한다.

- 마켓플레이스 (여러 판매자가 입점하는 오픈마켓 구조)
- 입점사 정산 (마켓플레이스 전제 기능)
- 라이브커머스
- 다국어 지원
- 다통화 지원

> ➕ 제안: 원래 "2차 고도화" 후보로 주셨던 목록 중 위 5개 항목을 "제외 범위"로 재분류했습니다. 이유는 단순 기능 추가가 아니라 서비스 구조 자체(단일 판매자 → 다수 판매자, 단일 통화 → 다통화)를 바꾸는 작업이라, MVP 이후 자연스러운 확장으로 보기 어렵기 때문입니다.

---

## 5. 사용자 유형

| 유형 | 설명 |
|---|---|
| 일반 고객 | 회원가입, 상품 탐색, 주문, 결제를 진행하는 최종 사용자 |
| 관리자 | 시스템 전체를 관리하는 최고 권한자 (계정 관리, 정책 설정 등) |
| 운영자 | 상품 등록, 주문 처리, 고객 대응 등 일상 업무를 수행하는 실무자 |

**➕ 제안**: MVP 단계에서는 관리자와 운영자를 **하나의 "관리자 계정"으로 통합**해서 시작하는 것을 권장한다. 두 역할의 권한을 세분화하는 작업(운영자는 상품/주문만 접근, 관리자는 전체 접근 등)은 그 자체로 복잡도가 있으므로 2차 고도화로 미루고, MVP에서는 "관리자 = 관리자+운영자 통합 권한"으로 단순화한다.

## 6. 핵심 업무 흐름

### 6-1. 고객 흐름
```
회원가입 → 로그인 → 상품 탐색(목록/상세/카테고리) → 장바구니 담기
→ 주문 생성 🔴 → 배송지 입력 → 결제 요청/승인 🔴 → 재고 차감 🔴
→ 주문 완료 → 주문 조회 → (배송 상태 확인)
```

### 6-2. 관리자 흐름
```
관리자 로그인 → 상품 등록/수정/재고 관리 → 주문 목록 확인
→ 주문 상태 변경(배송 준비/배송중/배송완료) 🔴
```

각 흐름 설명:

| 업무 | 설명 |
|---|---|
| 회원가입/로그인 | 고객이 계정을 만들고 서비스를 이용할 수 있는 상태가 되는 과정 |
| 상품 탐색 | 고객이 카테고리/목록/상세 페이지를 통해 상품 정보를 확인하는 과정 |
| 장바구니 | 구매 전 상품을 임시로 담아두는 기능 |
| 주문 🔴 | 장바구니의 상품을 실제 구매 의사로 확정하는 과정. 재고/결제와 직접 연결됨 |
| 결제 🔴 | 실제 금전 거래가 발생하는 과정. 승인/실패/중복 처리에 각별한 주의 필요 |
| 배송 | 결제 완료 후 상품이 고객에게 전달되는 과정 (MVP는 수동 상태 변경) |
| 관리자 상품 관리 | 상품 등록/수정/재고 수량 관리 |
| 관리자 주문 관리 | 주문 확인 및 처리 상태 변경 |

## 7. MVP 성공 기준

MVP는 아래 기준을 모두 만족했을 때 "완료"로 본다.

1. **기능 기준**: 회원가입부터 상품 탐색 → 장바구니 → 주문 → 결제 → 관리자 주문 확인까지 전체 흐름이 오류 없이 끝까지 동작한다.
2. **결제 기준**: 실제(또는 테스트) 결제가 최소 1건 이상 정상적으로 승인 처리되고, 결제 실패 시에도 주문이 이상한 상태로 남지 않는다.
3. **재고 기준**: 동시에 여러 주문이 들어와도 재고가 음수가 되거나 중복 판매(오버셀)가 발생하지 않는다.
4. **운영 기준**: 개발자의 도움 없이 관리자(기획자/운영자)가 상품 등록과 주문 확인/상태 변경을 화면에서 직접 수행할 수 있다.
5. **검수 기준**: 각 기능에 대해 비전문가도 따라 할 수 있는 검수 가이드로 직접 확인하고 승인했다.

## 8. AI 개발 시 주의해야 할 복잡한 영역

아래 영역은 겉으로 보기엔 간단해 보여도 실제로는 예외 처리가 많고, AI가 구현할 때 놓치기 쉬운 부분이다. 이 영역은 특히 꼼꼼한 계획·리뷰·테스트가 필요하다.

| 영역 | 왜 위험한가 |
|---|---|
| 재고 차감 🔴 | 동시에 여러 명이 같은 상품을 주문할 때 재고가 실제보다 더 많이 팔리는 "오버셀"이 발생할 수 있음. 단순히 "재고에서 빼기" 로직만으로는 부족하고, 동시 요청을 안전하게 처리하는 방법이 필요함 |
| 결제 승인/실패 🔴 | 결제가 승인됐는데 주문이 실패로 남거나, 반대로 결제가 실패했는데 주문은 성공으로 남는 "불일치 상태"가 생길 수 있음. 같은 결제 요청이 중복으로 두 번 승인되는 것도 막아야 함 |
| 주문 상태 관리 🔴 | 주문은 "결제대기 → 결제완료 → 배송준비 → 배송중 → 배송완료 → (취소/환불)"처럼 여러 상태를 오가는데, 잘못된 순서로 상태가 바뀌는 것을 막아야 함 (예: 배송완료된 주문을 다시 결제대기로 되돌리는 등) |
| 결제 취소/주문 취소 연동 | 주문을 취소했을 때 결제도 함께 취소(환불)되어야 하는데, 이 둘이 따로 놀면 고객은 환불을 못 받았는데 주문은 취소된 상태가 될 수 있음 |
| 관리자 권한 체크 | 관리자 화면의 기능을 고객 계정으로 접근하지 못하도록 막는 부분을 놓치면 보안 문제로 이어짐 |

> 위 표에 🔴로 표시된 항목은 이번 문서의 "위험 영역"과 동일하다. 이 영역의 설계/구현은 반드시 사전 설계 문서 리뷰와 별도 테스트 시나리오를 거친 뒤 진행한다. (세부 실행 지침은 별도 작업 원칙 문서의 민감 영역 처리 지침을 따른다.)

## 부록: 위험 영역 요약

🔴 **주문 / 결제 / 재고**는 이 프로젝트에서 가장 신중하게 다뤄야 할 세 가지 영역이다.

- 설계 단계에서 반드시 "이상 상황(동시 주문, 결제 실패, 중복 요청 등)"에 대한 대응 방식을 문서로 먼저 정리하고 승인받는다.
- 구현 후에는 정상 케이스뿐 아니라 이상 케이스까지 테스트를 거친 뒤에만 "완료"로 인정한다.
- 검수 시에는 실제로 이상 상황(예: 재고 1개 남은 상품을 두 명이 동시에 주문)을 재현해보는 것을 권장한다.
