# 화면 명세 (Screen Spec)

> 상태: 초안 (Draft) — 설계 승인자(기획자) 검토 필요
> 관련 문서: `docs/05_screen/screen-list.md`(화면 코드/경로), `docs/03_api/api-list.md`(호출 API), `docs/02_domain/domain-model.md`(상태값/비즈니스 규칙)
> 표시 규칙: 🔴 = 주문/결제/재고 관련 위험 화면, 🔒 = 로그인 필요, 🔒🔒 = 관리자 권한 필요

각 화면은 **목적 → 주요 사용자 → 진입 경로 → 와이어프레임 → 주요 UI 영역 → 입력 항목 → 버튼/액션 → 호출 API → Validation → 에러 케이스 → 비전문가 검수 포인트** 순서로 정리했다.

# Part 1. 사용자 화면 (`apps/web`)

## USR_HOME — 메인 화면

- **화면 목적**: 서비스 첫 진입 지점. 카테고리/신상품 등으로 탐색을 유도한다.
- **주요 사용자**: 비로그인·로그인 고객 모두
- **진입 경로**: URL 직접 접속, 로고 클릭, 로그인 후 리다이렉트

```
+----------------------------------------------------+
| [로고]      [검색창....................] [로그인][🛒] |
+----------------------------------------------------+
| [카테고리1] [카테고리2] [카테고리3] [카테고리4]         |
+----------------------------------------------------+
|                 [ 메인 배너 이미지 ]                  |
+----------------------------------------------------+
|  신상품                                              |
|  [상품카드][상품카드][상품카드][상품카드]                |
+----------------------------------------------------+
| [푸터: 이용약관 | 개인정보처리방침 | 고객센터]           |
+----------------------------------------------------+
```

- **주요 UI 영역**: 헤더(로고/검색/로그인·마이페이지/장바구니), 카테고리 바로가기, 메인 배너, 신상품 목록, 푸터
- **입력 항목**: 검색어(선택)
- **버튼/액션**: 검색 실행, 카테고리 클릭(→`USR_PRODUCT_LIST`), 상품카드 클릭(→`USR_PRODUCT_DETAIL`), 로그인/마이페이지/장바구니 아이콘 클릭
- **호출 API**: `GET /api/v1/categories`, `GET /api/v1/products`(신상품 노출용, `sort=createdAt:desc`)
- **Validation**: 검색어 앞뒤 공백 제거, 빈 검색어는 검색 미실행
- **에러 케이스**: 상품/카테고리 로딩 실패 시 빈 화면 대신 "일시적 오류" 안내와 새로고침 버튼 표시
- **비전문가 검수 포인트**: 로그인 여부에 따라 헤더 우측 아이콘(로그인 버튼 ↔ 마이페이지/장바구니 아이콘)이 올바르게 바뀌는지, 카테고리 클릭 시 해당 카테고리 상품만 정확히 나오는지

## USR_LOGIN — 로그인 화면

- **화면 목적**: 회원 로그인
- **주요 사용자**: 비로그인 고객
- **진입 경로**: 헤더 로그인 버튼, 로그인이 필요한 화면에 비로그인 상태로 접근 시 자동 이동

```
+------------------------------+
|            로그인             |
+------------------------------+
| 이메일    [__________________] |
| 비밀번호  [__________________] |
|                                |
|         [   로그인   ]         |
|                                |
| 회원가입        비밀번호를 잊으셨나요? |
+------------------------------+
```

- **주요 UI 영역**: 이메일/비밀번호 입력폼, 로그인 버튼, 회원가입·비밀번호 재설정 링크
- **입력 항목**: `email`, `password`
- **버튼/액션**: 로그인 제출, 회원가입 이동(→`USR_SIGNUP`), 비밀번호 재설정 요청(별도 모달 또는 화면 — ➕ 확인 필요: 이번 화면 목록에 비밀번호 재설정 전용 화면이 없어 모달 처리로 가정, 별도 화면 필요 여부 확인 필요)
- **호출 API**: `POST /api/v1/auth/login`
- **Validation**: 이메일 형식 검증, 두 필드 모두 필수
- **에러 케이스**: 자격증명 불일치(`401`), 계정 잠금(`403`), 탈퇴 계정(`403`)
- **비전문가 검수 포인트**: 이메일이 틀렸는지 비밀번호가 틀렸는지 **구체적으로 알려주지 않고** "이메일 또는 비밀번호를 확인해주세요"처럼 뭉뚱그려 안내하는지 (계정 존재 여부 노출 방지 — 보안 관례)

## USR_SIGNUP — 회원가입 화면

- **화면 목적**: 신규 회원가입
- **주요 사용자**: 비로그인 고객
- **진입 경로**: 로그인 화면 링크

```
+------------------------------+
|           회원가입             |
+------------------------------+
| 이메일       [______________] |
| 비밀번호     [______________] |
| 비밀번호 확인 [______________] |
| 이름        [______________]  |
| 연락처      [______________]  |
| [ ] 이용약관 및 개인정보처리방침 동의 |
|                                |
|         [  가입하기  ]         |
+------------------------------+
```

- **주요 UI 영역**: 입력 폼, 약관 동의 체크박스, 가입 버튼
- **입력 항목**: `email`, `password`, `passwordConfirm`, `name`, `phoneNumber`, 약관 동의 여부
- **버튼/액션**: 가입하기 제출, 로그인 화면으로 이동
- **호출 API**: `POST /api/v1/auth/signup`
- **Validation**: 이메일 형식, 비밀번호 최소 길이/조합 규칙(➕ 확인 필요 — 구체적 정책 미정), 비밀번호=비밀번호 확인 일치, 약관 동의 필수(미동의 시 가입 버튼 비활성화)
- **에러 케이스**: 이메일 중복(`409`), 입력값 오류(`400`)
- **비전문가 검수 포인트**: 약관 미동의 상태에서 가입 버튼이 실제로 눌리지 않는지, 이메일 중복 시 "이미 가입된 이메일입니다"처럼 명확히 안내되는지

## USR_PRODUCT_LIST — 상품 목록 화면

- **화면 목적**: 카테고리 탐색 또는 검색 결과로 상품을 나열해 보여준다.
- **주요 사용자**: 비로그인·로그인 고객 모두
- **진입 경로**: 헤더 검색, 카테고리 클릭, 메인 화면 "더보기"

```
+----------------------------------------------------+
| [카테고리 필터] [가격범위] [정렬: 최신순 v]            |
+----------------------------------------------------+
| [상품][상품][상품][상품]                              |
| [상품][상품][상품][상품]                              |
+----------------------------------------------------+
|              [ 1 2 3 ... 다음 ]                      |
+----------------------------------------------------+
```

- **주요 UI 영역**: 필터바(카테고리/가격/정렬), 상품 그리드, 페이지네이션
- **입력 항목**: `keyword`, `categoryId`, `minPrice`, `maxPrice`, `sort`, `page`
- **버튼/액션**: 필터 적용, 정렬 변경, 페이지 이동, 상품카드 클릭(→`USR_PRODUCT_DETAIL`)
- **호출 API**: `GET /api/v1/products`, `GET /api/v1/categories`(필터용)
- **Validation**: `minPrice` ≤ `maxPrice`
- **에러 케이스**: 검색 결과 없음(빈 상태 안내 문구), API 오류 시 재시도 버튼 노출
- **비전문가 검수 포인트**: 품절 옵션만 남은 상품도 목록에는 그대로 노출되고 "품절" 뱃지가 붙는지(상품 자체는 숨김 처리 대상이 아님), 페이지당 노출 개수가 `api-convention.md` 기본값(20)과 일치하는지

## USR_PRODUCT_DETAIL — 상품 상세 화면

- **화면 목적**: 상품 상세 정보 확인, 옵션 선택 후 장바구니 담기/바로구매
- **주요 사용자**: 비로그인·로그인 고객 모두 (구매 액션은 로그인 필요)
- **진입 경로**: 목록/검색결과 클릭

```
+----------------------------------------------------+
| [이미지 갤러리]        | 상품명                       |
| [썸네일][썸네일][썸네일] | 가격: 00,000원               |
|                        | 옵션: [드롭다운: 색상/사이즈] |
|                        | 수량: [ - 1 + ]              |
|                        | [ 장바구니 담기 ] [ 바로구매 ] |
+----------------------------------------------------+
| [상세설명 / 리뷰 탭]                                 |
+----------------------------------------------------+
```

- **주요 UI 영역**: 이미지 갤러리, 상품 정보, 옵션 선택, 수량 선택, 액션 버튼, 상세설명 영역
- **입력 항목**: `productOptionId`(옵션 선택), `quantity`
- **버튼/액션**: 이미지 확대, 장바구니 담기, 바로구매(→`USR_ORDER_FORM` 직행)
- **호출 API**: `GET /api/v1/products/{productId}`, `GET /api/v1/products/{productId}/options`, (담기) `POST /api/v1/cart/items`
- **Validation**: 옵션 미선택 시 액션 버튼 비활성화, 수량은 1 이상 정수
- **에러 케이스**: 품절 옵션 선택 시 안내 및 버튼 비활성화, 비로그인 상태에서 담기/구매 시도 시 로그인 화면으로 안내
- **비전문가 검수 포인트**: 옵션별 추가금액이 최종 가격에 정확히 더해지는지, 품절 옵션은 드롭다운에서 선택 자체가 안 되거나 "품절" 표시로 구분되는지

## USR_CART 🔒 — 장바구니 화면

- **화면 목적**: 담아둔 상품을 확인·수정하고 주문 단계로 진행
- **주요 사용자**: 로그인 고객
- **진입 경로**: 헤더 장바구니 아이콘, 상품 상세에서 "담기" 후

```
+----------------------------------------------------+
| [이미지] 상품명 / 옵션명   [ - 수량 + ]  00,000원 [삭제] |
| [이미지] 상품명 / 옵션명   [ - 수량 + ]  00,000원 [삭제] |
+----------------------------------------------------+
|                                   합계: 000,000원    |
|                                   [   주문하기   ]    |
+----------------------------------------------------+
```

- **주요 UI 영역**: 항목 리스트(이미지/옵션/수량/가격/삭제), 합계 영역, 주문하기 버튼
- **입력 항목**: 수량 변경값
- **버튼/액션**: 수량 증감, 항목 삭제, 전체 비우기, 주문하기(→`USR_ORDER_FORM`)
- **호출 API**: `GET /api/v1/cart`, `PATCH /api/v1/cart/items/{cartItemId}`, `DELETE /api/v1/cart/items/{cartItemId}`, `DELETE /api/v1/cart`
- **Validation**: 수량 최소 1, 빈 장바구니 상태에서는 "주문하기" 버튼 비활성화
- **에러 케이스**: 담긴 옵션이 이후 품절/숨김 처리된 경우 항목에 "품절" 표시(➕ 확인 필요: 이 경우 자동 제외 여부 등 구체적 UX 정책 미정 — 실제 품절 여부의 최종 확인은 주문 생성 시점에 이루어짐, `domain-model.md` Cart 규칙)
- **비전문가 검수 포인트**: 합계 금액이 옵션 추가금액까지 포함해 정확히 계산되는지, 품절 항목이 있어도 나머지 항목만으로 주문을 진행할 수 있는지(➕ 확인 필요 사항과 연결)

## USR_ORDER_FORM 🔒🔴 — 주문서 화면

- **화면 목적**: 배송지 입력과 결제수단 선택 후 주문 생성 및 결제 요청을 시작한다. **위험 화면 — 재고 예약과 결제가 이 화면에서 시작됨**
- **주요 사용자**: 로그인 고객
- **진입 경로**: 장바구니 "주문하기", 상품 상세 "바로구매"

```
+----------------------------------------------------+
| 주문 상품                                            |
| [이미지] 상품명 / 옵션 / 수량        00,000원         |
+----------------------------------------------------+
| 배송지                                               |
| ( ) 등록된 배송지 선택 [드롭다운]                     |
| ( ) 새 배송지 입력 [폼: 수령인/연락처/주소]            |
+----------------------------------------------------+
| 결제수단  [카드] [계좌이체] ...                       |
+----------------------------------------------------+
|                              최종 결제금액: 000,000원 |
|                              [     결제하기     ]     |
+----------------------------------------------------+
```

- **주요 UI 영역**: 주문상품 요약, 배송지 선택/입력, 결제수단 선택, 최종금액, 결제하기 버튼
- **입력 항목**: `addressId`(선택) 또는 직접입력(`recipientName`,`phoneNumber`,`postalCode`,`addressLine1`,`addressLine2`), `paymentMethod`
- **버튼/액션**: 새 배송지 추가, 결제하기(제출 — 클릭 즉시 비활성화 처리 필요, 아래 검수 포인트 참고)
- **호출 API**: `GET /api/v1/me/addresses`, `POST /api/v1/orders` 🔴, `POST /api/v1/orders/{orderId}/payments` 🔴
- **Validation**: 배송지 필수 입력, 결제수단 선택 필수
- **에러 케이스**: 재고 부족(`409`, 어떤 상품이 문제인지 구체적으로 안내 후 장바구니로 복귀 유도), 배송지 누락(`400`), 결제 요청 자체가 실패(네트워크 오류 등)한 경우 재시도 안내
- **비전문가 검수 포인트**: **"결제하기" 버튼을 연속으로 여러 번 눌러도 주문/결제 요청이 중복으로 나가지 않는지** (버튼 클릭 즉시 비활성화 또는 로딩 처리 여부 확인 — 서버의 멱등키 처리와 별개로 화면에서도 1차 방어가 되어야 함), 재고 부족 시 "어떤 상품의 재고가 부족한지" 화면에 명확히 표시되는지

## USR_PAYMENT_RESULT 🔒🔴 — 결제 결과 화면

- **화면 목적**: 결제 성공/실패 결과를 안내한다.
- **주요 사용자**: 로그인 고객
- **진입 경로**: PG 결제창에서 결제 완료 후 리다이렉트

```
+----------------------------------------+
|              [ 성공 아이콘 ]              |
|         결제가 완료되었습니다             |
|          주문번호: 20260707-0001        |
|                                          |
|            [ 주문내역 보기 ]              |
+----------------------------------------+
```

- **주요 UI 영역**: 결과 아이콘, 안내 문구, 주문번호, 액션 버튼
- **입력 항목**: 없음
- **버튼/액션**: 성공 시 "주문내역 보기"(→`USR_ORDER_DETAIL`), 실패 시 "다시 시도"(→재결제 흐름, `POST /api/v1/orders/{orderId}/retry-payment`)
- **호출 API**: `GET /api/v1/orders/{orderId}`(최신 주문 상태 확인)
- **Validation**: 없음
- **에러 케이스**: 콜백 처리가 아직 끝나지 않아 상태가 `PENDING`으로 남아있는 경우 "결제 확인 중" 안내(➕ 확인 필요: 자동 새로고침/폴링 방식 및 간격 미정), 존재하지 않는 주문 접근(`404`)
- **비전문가 검수 포인트**: **이 화면의 성공/실패 표시는 반드시 서버의 주문 상태 조회 API 결과를 기준으로 판단해야 하며, PG가 브라우저에 넘겨주는 파라미터만 보고 화면에서 임의로 "성공"이라 표시하면 안 된다** (`domain-model.md` 8장 "결제 결과의 최종 확정 기준은 PG 서버 콜백" 원칙이 화면에도 그대로 적용되어야 함 — 이 화면 검수 시 가장 중요하게 확인할 부분)

- **비전문가 검수 포인트**: 소속 상품이 있는 카테고리를 삭제하려 할 때 경고 없이 바로 삭제되지 않는지 (정책 미확정 상태에서 사고를 막는 최소 안전장치)

## ADM_ORDER_LIST 🔒🔒🔴 — 주문 목록

- **화면 목적**: 전체 회원의 주문을 상태/기간별로 조회
- **주요 사용자**: 관리자
- **진입 경로**: 대시보드, 좌측 네비게이션

```
+----------------------------------------------------+

+----------------------------------------------------+
| 주문번호 | 회원 | 금액 | 상태 | 주문일시               |
+----------------------------------------------------+
|              [ 1 2 3 ... 다음 ]                      |
+----------------------------------------------------+
```

- **주요 UI 영역**: 필터바, 주문 테이블, 페이지네이션
- **입력 항목**: `status`, `fromDate`, `toDate`, `page`
- **버튼/액션**: 필터 적용, 행 클릭(→`ADM_ORDER_DETAIL`)
- **호출 API**: `GET /api/v1/admin/orders`
- **Validation**: `fromDate` ≤ `toDate`
- **에러 케이스**: 없음(결과 없으면 빈 목록)
- **비전문가 검수 포인트**: 결제실패가 반복되는 등 주의가 필요한 주문이 눈에 띄게 표시되는지(➕ UX 제안, 필수 아님)

## ADM_ORDER_DETAIL 🔒🔒🔴 — 주문 상세

- **화면 목적**: 개별 주문 확인 및 상태변경/취소/환불/배송 처리
- **주요 사용자**: 관리자
- **진입 경로**: 주문 목록 행 클릭

```
+----------------------------------------------------+
| 주문번호 20260707-0001         상태: [드롭다운: 다음 상태로 변경] |
+----------------------------------------------------+
| 주문상품 / 배송지 / 결제이력(재시도 포함) 표시           |
+----------------------------------------------------+
| 배송: [배송 시작 처리] (운송장번호 입력) / [배송완료 처리] |
+----------------------------------------------------+
|                 [ 주문 취소(환불) ]                   |
+----------------------------------------------------+
```

- **주요 UI 영역**: 주문 정보, 상태 변경 컨트롤, 배송 처리 버튼, 취소(환불) 버튼
- **입력 항목**: 다음 상태(드롭다운, 허용된 전이만 선택 가능), 운송장번호/택배사명, 취소 사유
- **버튼/액션**: 상태 변경, 배송 시작/완료 처리, 주문 취소(환불)
- **호출 API**: `GET /api/v1/admin/orders/{orderId}`, `PATCH /api/v1/admin/orders/{orderId}/status`, `POST /api/v1/admin/orders/{orderId}/cancel` 🔴, `PATCH /api/v1/admin/shipments/{shipmentId}/start`, `PATCH /api/v1/admin/shipments/{shipmentId}/complete`
- **Validation**: 상태 드롭다운은 **현재 상태에서 허용되는 다음 상태만 선택지로 보여준다** (`domain-model.md` 3장 상태 전이표 기준. 허용되지 않는 상태는 아예 드롭다운에 나타나지 않아야 한다)
- **에러 케이스**: 허용되지 않는 상태 전환 시도(`409`), 배송완료 후 취소 시도(`409`)
- **비전문가 검수 포인트**: **주문 취소(환불) 버튼을 누르면 반드시 "정말 취소하시겠습니까? 환불이 함께 처리됩니다" 같은 확인 모달이 뜨는지** — 되돌리기 어려운 가장 민감한 동작이므로 이 화면에서 가장 중요하게 확인해야 할 부분. 상태 변경 드롭다운에 허용되지 않는 상태가 보이지 않는지도 함께 확인.

- **주요 사용자**: 관리자
- **진입 경로**: 좌측 네비게이션

- **버튼/액션**: 검색, 행 클릭(회원 상세 — 이번 화면 목록에는 상세 화면이 별도로 없어 목록 내 요약 확인 수준으로 가정, ➕ 확인 필요: 회원 상세 화면 필요 여부)

- **비전문가 검수 포인트**: 비밀번호 해시 등 민감정보가 목록 어디에도 노출되지 않는지

## ADM_INVENTORY_MANAGE 🔒🔒🔴 — 재고 관리

- **화면 목적**: 옵션별 재고 현황 확인 및 수량 조정(입고 등)
- **주요 사용자**: 관리자
- **진입 경로**: 대시보드, 상품 수정 화면의 재고 링크

```
+----------------------------------------------------+
| [검색: 상품명/옵션명]                                |
+----------------------------------------------------+
| 상품명 | 옵션명 | 총재고 | 예약중 | 가용재고 | [조정]  |
+----------------------------------------------------+
```

- **주요 UI 영역**: 검색, 재고 테이블, 조정 입력
- **입력 항목**: `keyword`, 조정 수량(`quantityChange`)
- **버튼/액션**: 조정 저장(확인 모달 필요)
- **호출 API**: `GET /api/v1/admin/inventory` (✅ 신규 추가됨), `PATCH /api/v1/admin/inventory/{optionId}`(조정)
- **Validation**: 조정 후 총재고가 예약중 수량보다 작아지지 않아야 함
- **에러 케이스**: 조정 결과 재고가 음수가 되는 경우(`400`), 예약중 수량보다 적게 낮추려는 시도(`409 INVENTORY_ADJUSTMENT_CONFLICT`)
- **비전문가 검수 포인트**: 예약중인 수량이 있는 옵션의 재고를 함부로 낮추려 할 때 명확한 경고와 함께 저장이 막히는지 — 이 화면에서 가장 위험한 실수 지점
