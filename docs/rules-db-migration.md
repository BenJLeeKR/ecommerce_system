# DB / 마이그레이션 세부 원칙

> 이 문서는 `CLAUDE.md` 절대 규칙 4의 실행 지침이다. DB 스키마 변경이 포함된 작업을 할 때만 참조한다.

## 사전 보고 항목

스키마 변경(테이블/컬럼 추가·삭제·타입 변경, 인덱스 변경 포함)은 **적용 전** 다음을 보고한다.

1. 변경 스키마 diff
2. 기존 데이터에 미치는 영향 (NULL 처리, 기본값, 대용량 테이블 락 여부)
3. 롤백 마이그레이션 스크립트 존재 여부

## 원칙

- 마이그레이션은 항상 **되돌릴 수 있는 형태(down migration)** 로 작성한다. 불가능하면 이유와 대안(백업 시점 등)을 명시한다.
- 운영 데이터가 있는 환경에서는 마이그레이션 실행 전 백업 여부를 확인 문구로 남긴다.
- 파괴적 변경(컬럼/테이블 삭제)은 즉시 삭제하지 않고, 2단계로 진행한다.
  1. 1단계: 애플리케이션 코드에서 참조 제거 + 배포 + 일정 기간 모니터링
  2. 2단계: 확인 후 별도 이슈로 실제 컬럼/테이블 삭제

## 체크리스트

- [ ] 스키마 diff를 문서로 제시했는가
- [ ] down migration이 존재하는가 (없다면 대안을 명시했는가)
- [ ] 대용량 테이블 대상 시 락/성능 영향을 검토했는가
- [ ] 백업 계획을 확인했는가
- [ ] 파괴적 변경은 2단계 방식을 따르는가
